# حالة القسم الثاني عشر: نظام إدارة المخططات الهندسية (Drawings Management)

## ما تم إنجازه في هذه الجلسة

### 1) إصلاح حرِج: الجزء 8/10 — إدارة الاعتمادات المتعددة (كان تالفاً بالكامل)

اكتُشف أن `backend/utils/drawingApprovals.js` من تسليم سابق كان يحتوي **كود واجهة
أمامية (`document.`/`window.`) بالخطأ بدل منطق الباكيند**، وبلا `module.exports`
إطلاقاً. النتيجة: جميع نداءات `DRAW_APPROVALS.*` داخل `server.js` (كل endpoints
الاعتمادات الثمانية) كانت ستفشل فوراً عند التشغيل الفعلي رغم ظهورها "مكتملة" في
الكود الظاهري.

تم إعادة كتابة الملف بالكامل كمنطق باكيند حقيقي فوق نفس طبقة `drawings.json`:

| الوظيفة | الوصف |
|---|---|
| `defineApprovalRequirement` | تحديد مستويات الاعتماد المطلوبة لكل مخطط (ترتيب مرجعي ثابت) |
| `getApprovalStatusInfo` | حالة الاعتماد الكلية: المكتمل/المتبقي/المستوى التالي |
| `submitApproval` | تسجيل اعتماد أو رفض بتوقيع إلكتروني (بصمة SHA-256 من محتوى القرار + الإصدار وقت التوقيع + الوقت) |
| `revokeApproval` | إلغاء اعتماد سابق مع سبب، وإعادة تقييم حالة المخطط تلقائياً |
| `listApprovals` | سجل كامل لاعتمادات مخطط |
| `verifyApproval` | تحقق فعلي من سلامة التوقيع (سليم/ملغى/تعديل لاحق للمخطط/متلاعب به) |
| `getApprovalCountForDashboard` | عدّاد فعلي للوحة التحكم الرئيسية |

**تحقق فعلي عبر HTTP end-to-end (وليس اختبار وحدة معزول):** إنشاء مخطط → تحديد
مستويين (داخلي + نهائي) → توقيع الأول → توقيع الثاني → التحقق من اكتمال الاعتماد
تلقائياً وتحديث `approval_status = approved` و`approver` على سجل المخطط نفسه. كل
خطوة اختُبرت عبر طلبات HTTP فعلية على خادم يعمل، وليس استدعاء دوال مباشرة فقط.

### 2) الجزء 9/10 — التكامل مع BIM + اكتشاف تعارضات وصفي (جديد بالكامل)

ملف جديد `backend/utils/drawingBIM.js` يبني فوق تصنيف `bim` الموجود مسبقاً في
`drawingManagement.js` (الجزء 1/10):

| البند | التفاصيل |
|---|---|
| ربط مخطط بمصدر BIM | Revit / IFC / Navisworks / Civil 3D / Tekla Structures، مع بيانات استيراد فعلية (اسم الملف، حجمه) |
| تسجيل عناصر النموذج | كل عنصر بموقعه المكاني الحقيقي (bounding box ثلاثي الأبعاد X/Y/Z) ونظامه/تخصصه، فردياً أو دفعة واحدة (bulk) |
| ربط عناصر النموذج بالمخططات | ربط عنصر BIM ثلاثي الأبعاد بمخطط 2D تفصيلي (كما ورد حرفياً في المتطلبات) |
| اكتشاف التعارضات (Clash Detection) | خوارزمية تقاطع صناديق محيطة حقيقية (AABB Intersection) بين عناصر من تخصصات مختلفة، بهامش تسامح قابل للتخصيص |
| تصنيف شدة التعارض | حرِج / متوسط / بسيط، محسوبة فعلياً من حجم التداخل بالمتر المكعب |
| مزامنة البيانات | إعادة تشغيل الفحص يُحدّث التعارضات القائمة بدل تكرارها، ويعيد فتح أي تعارض "محلول" ظهر مجدداً |
| إدارة حالة التعارض | مفتوح / قيد المعالجة / محلول / متجاهَل + ملاحظة حل |
| سجل تشغيلات الفحص | تتبّع كل عملية اكتشاف تعارضات (عدد العناصر المفحوصة، عدد التعارضات، من شغّلها، متى) |
| لوحة ملخص BIM | إحصائيات فعلية لكل مخطط: عدد العناصر حسب النظام، التعارضات المفتوحة/الحرجة/المحلولة |

**تحقق حسابي دقيق:** عمود إنشائي (صندوق 0,0,0 → 1,1,3) ومجرى تكييف متقاطع جزئياً
(صندوق 0.5,0.2,1 → 2,0.8,1.5) → حجم التداخل المحسوب فعلياً = **0.15 م³** مطابق
تماماً للحساب اليدوي (0.5 × 0.6 × 0.5). عنصر ثالث بعيد تماماً (كمرة عند
10,10,10) لم يُكتشف له أي تعارض، كما هو متوقع هندسياً.

**تحقق عبر HTTP end-to-end:** إنشاء مخطط BIM → ربط بمصدر Revit → إضافة عنصرين
متقاطعين → تشغيل اكتشاف التعارضات → التحقق من ظهور التعارض بالحجم الصحيح في كل
من `/bim/clashes/list` و`/bim/summary` و`/api/drawings/dashboard` (عبر عدّاد
`open_bim_clashes` الجديد المُدمَج في لوحة التحكم الرئيسية).

13 endpoint جديد مُضافة في `server.js` تحت `/api/drawings/bim/*`، جميعها محمية
بنفس نظام الصلاحيات (`requirePermission(req, 'drawings', ...)`) المستخدم في بقية
أجزاء القسم.

| # | الجزء | الحالة |

| 2/10 | إدارة الإصدارات المتقدمة | ✅ منجَز |
| 3/10 | عارض المخططات (قياسات/إحداثيات/فتح متعدد) | ✅ منجَز |
| 4/10 | إدارة الطبقات (Layers) الوصفية | ✅ منجَز |

راجع `DRAWINGS_PLAN.md` المرفَق للتفاصيل الكاملة لكل جزء منجَز ومتبقٍ.

عند رغبتك بمتابعة الجزء الأخير (10/10 — التنبيهات + التقارير + الذكاء الاصطناعي
+ التكامل الشامل)، أخبرني وسأنفذه بنفس الأسلوب: فحص كامل للكود القائم أولاً، ثم
كود حقيقي فعلي، ثم اختبار HTTP end-to-end فعلي، ثم تسليم بنفس هذه الصيغة.

## ملاحظة مهمة للمتابعة المستقبلية

نظراً لاكتشاف ملف تالف واحد (`drawingApprovals.js`) في تسليم سابق دون أن يظهر ذلك
في حينه، يُنصح عند أي جلسة قادمة بالتحقق دورياً من أن كل ملفات `backend/utils/`
الخاصة بالقسم تحتوي فعلياً على `module.exports` صحيحاً ولا تحتوي أي إشارات
`document.`/`window.` (وهو فحص سريع نُفِّذ في هذه الجلسة على كل ملفات المخططات
الثمانية القائمة ولم يظهر فيها خلل آخر).
