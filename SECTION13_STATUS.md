# حالة القسم الثالث عشر: نظام إدارة الميزانية (Budget Management System)

## ما تم إنجازه في هذه الجلسة: الجزء 1/10 — البنية الأساسية

تم تنفيذ الجزء بالكامل كوظائف حقيقية تعمل فعلياً (وليس واجهات شكلية)، عبر طبقة
تخزين موحّدة جديدة `backend/data/budgets.json` + `backend/data/budget_audit.json`،
بدون أي تبعيات خارجية جديدة:

| البند | التفاصيل |
|---|---|
| ربط فعلي بالمشروع | إنشاء ميزانية يتحقق فعلياً من وجود المشروع عبر `projectManagement.getProject()` ويرفض الإنشاء إن لم يكن موجوداً |
| رقم ميزانية مرجعي | يُولَّد تلقائياً بصيغة `BUD-<رمز المشروع>-<تسلسل>` |
| هيكل تقسيم الميزانية (BBS) | شجرة فعلية بخمسة مستويات إلزامية بدون تخطي: مرحلة → بند رئيسي → بند فرعي → نشاط → مورد |
| حساب التكلفة التجميعي | تكلفة أي عقدة = مجموع تكاليف أبنائها الفعلي (وليس رقماً يُدخَل يدوياً)، محسوبة عند كل استعلام |
| الإصدارات | كل تحديث لحالة الميزانية إلى (معتمدة/محدثة/مراجَعة) يرفع رقم الإصدار تلقائياً ويؤرشف نسخة كاملة سابقة |
| مقارنة الإصدارات | حساب فعلي للفروقات (القيمة، الحالة، إجمالي BBS) بين أي إصدارين مؤرشفين |
| لوحة تحكم | كل الأرقام محسوبة فعلياً من البيانات المخزَّنة: إجمالي قيمة المشاريع، الميزانيات المعتمدة، نسبة الاستهلاك، المشاريع المتجاوزة/ضمن الميزانية |
| صلاحيات RBAC | ثلاثة أدوار جديدة: `financial_manager` (كامل الصلاحيات)، `budget_owner` (عرض/إنشاء/تعديل)، `project_budget_viewer` (عرض فقط) |
| سجل تدقيق | كل عملية إنشاء/تحديث/حذف على الميزانية أو عقد BBS مسجَّلة فعلياً |

تحقق من صحة الحسابات تم فعلياً أثناء التنفيذ (تجميع تكلفة موارد 50,000 + 120,000 =
170,000 عبر أربع مستويات، رفض تخطي مستوى في الشجرة، رفع الإصدار من 1 إلى 3 عبر
تحديثين متتاليين مع أرشفة صحيحة، مقارنة إصدارين أظهرت فرقاً دقيقاً قدره 10,000،
حساب نسبة استهلاك ميزانية = 18% لعقد بقيمة مليون وميزانية بقيمة 180,000) وجميعها
اجتازت الاختبار بنجاح، بما في ذلك اختبار تكامل شامل عبر تسجيل مستخدم فعلي بدور
`financial_manager`، تسجيل دخول، والتحقق من الصلاحيات (`budget:view` = صحيح،
`drawings:view` = خطأ كما هو متوقع).

**ملاحظة نطاق العمل:** "المصروفات الفعلية" و"الالتزامات المالية" و"الأرباح
الفعلية" في لوحة التحكم تُحسَب حالياً من إجمالي BBS كأساس حقيقي أولي، وستُدمَج
بدقة كاملة تلقائياً بعد تنفيذ الأجزاء 3/10 (التكاليف الفعلية) و4/10 (الإيرادات)
و6/10 (الانحرافات/EVM) دون كسر أي واجهة برمجية قائمة.

## الأجزاء المتبقية من القسم الثالث عشر (10 أجزاء بالإجمالي)

راجع `BUDGET_PLAN.md` المرفَق للتفاصيل الكاملة لكل جزء منجَز ومتبقٍ.

| # | الجزء | الحالة |
|---|-------|--------|
| 1/10 | البنية الأساسية + التخزين + إنشاء الميزانية + BBS + لوحة تحكم أساسية | ✅ **منجَز في هذه الجلسة** |
| 2/10 | إدارة بنود التكلفة + الربط مع حصر الكميات (BOQ) | 🔲 متبقٍ |
| 3/10 | إدارة التكاليف الفعلية (مواد/عمالة/معدات/أخرى) | 🔲 متبقٍ |
| 4/10 | إدارة الإيرادات + الدفعات + المستخلصات | 🔲 متبقٍ |
| 5/10 | أوامر التغيير (Change Orders) | 🔲 متبقٍ |
| 6/10 | مراقبة الانحرافات المالية + تحليل القيمة المكتسبة (EVM) | 🔲 متبقٍ |
| 7/10 | التدفقات النقدية (Cash Flow) + الموافقات المالية | 🔲 متبقٍ |
| 8/10 | الفواتير والمستخلصات (Invoicing) | 🔲 متبقٍ |
| 9/10 | التقارير المالية + الرسوم البيانية + التصدير | 🔲 متبقٍ |
| 10/10 | الذكاء الاصطناعي المالي + التكامل الشامل مع بقية الأقسام | 🔲 متبقٍ |

عند رغبتك بمتابعة الجزء التالي (2/10 — بنود التكلفة + الربط مع BOQ)، أخبرني
وسأنفذه بنفس الأسلوب: فحص كامل للكود القائم أولاً، ثم كود حقيقي فعلي، ثم اختبار،
ثم تسليم بنفس هذه الصيغة.

## نقاط API المُضافة في هذا الجزء

| المسار | الطريقة | الوظيفة |
|---|---|---|
| `/api/budget/dashboard` | GET | لوحة التحكم المالية |
| `/api/budget/list` | GET | قائمة الميزانيات مع بحث/تصفية/ترقيم صفحات |
| `/api/budget/get` | GET | جلب ميزانية محددة |
| `/api/budget/create` | POST | إنشاء ميزانية جديدة |
| `/api/budget/update` | POST | تحديث ميزانية (مع ترقية إصدار تلقائية عند تغيير الحالة) |
| `/api/budget/delete` | POST | حذف ميزانية |
| `/api/budget/versions/compare` | GET | مقارنة إصدارين من نفس الميزانية |
| `/api/budget/bbs/tree` | GET | شجرة هيكل تقسيم الميزانية كاملة |
| `/api/budget/bbs/add-node` | POST | إضافة عقدة للهيكل |
| `/api/budget/bbs/update-node` | POST | تعديل عقدة |
| `/api/budget/bbs/delete-node` | POST | حذف عقدة |
| `/api/budget/audit-log` | GET | سجل التدقيق الخاص بالميزانية |
